Transaction-Safety Sweep — Webroot Security + Atomic Money Writes

PRs #1078–#1083 · 7 commits · capstone test suite

The largest single body of work today. Started from a question: "is every multi-step money write in this system actually safe if it fails halfway?" The answer was no in several places, and the webroot itself had a much bigger problem — 284 unauthenticated scripts sitting at the site root. Both were closed out in one coordinated sweep across six branches, capped by an end-to-end integrity test that proves the four statutory reports still balance after all of it.

SECURITY
284 Unauthenticated Scripts Quarantined Out of the Public Webroot
Finding: Of 299 top-level .php files, only 15 are real entry points (verified against the roots.php router, every JS/fetch call, includes, CI and deploy). The other 284 were pre-migration-system dev/debug/data-fix scripts (check_*, debug_*, fix_*, alter_*...) reachable over plain HTTP with no authentication — several would run ALTER TABLE/UPDATE against the production database for anyone who hit the URL.
Fix: All 284 moved to scripts/legacy/ via git mv (full history preserved), slated for permanent deletion one stable release after merge. .htaccess now denies HTTP access to scripts/, tests/, cron/, migrations/ (except the login-guarded migrations/status.php dashboard). migrations/runner.php is now CLI-only (PHP_SAPI guard) — deploy over SSH unaffected.
Test: tests/test_webroot_quarantine_cli.php — 9/9 passing (webroot whitelist, .htaccess rules, runner CLI guard, ajax_* endpoints still authenticate).
Status: merged to develop (PR #1078) — not yet promoted to main.
FOUNDATION
80 Tables Silently Ignoring Rollbacks — Converted MyISAM → InnoDB
Bug: 80 tables (payroll, payment_vouchers, bank_transactions, pos_payments, product_stocks, loans, ...) were still MyISAM, a storage engine with no transaction support — every existing beginTransaction()/commit() touching them was doing nothing on failure.
Fix: migrations/2026_07_01_convert_myisam_to_innodb.php — criteria-based, idempotent, converts with ROW_FORMAT=DYNAMIC (fixes legacy row_format=FIXED tables that would otherwise fail under innodb_strict_mode). This underpins every atomicity fix below.
FIX
Payroll — Multi-Step Writes Made Atomic
Bug: Payroll payout/edit endpoints ran 3–5 dependent writes (status update → GL posting → bank register/accrual reversal → re-post) with no transaction — a mid-sequence failure could leave some employees paid with GL posted and others not, or an amount changed with its accrual reversed but never reposted.
Fix: api/bulk_update_payroll_status.php — each employee's payment is now one transaction per row; a failure rolls that row back completely and is reported in the response, other rows proceed; a null GL posting is now fatal for the row. api/update_payroll.php, api/operations/process_project_payroll.php similarly wrapped.
Test: tests/test_payroll_tx_atomicity_cli.php — 16 checks, all 8 existing payroll suites still pass.
FIX
IPC / Invoice / Voucher — Multi-Step Writes Made Atomic
Bug: Worst case found — IPC invoicing ran INSERT invoices then UPDATE interim_payment_certificates with no transaction. A failure between the two left an invoice with no IPC link, and a retry created a duplicate invoice for the same certificate.
Fix: api/operations/create_invoice_from_ipc.php — number allocation + invoice insert + IPC link are one transaction, link update guarded on invoice_id IS NULL so two concurrent requests can't both invoice the same certificate. update_ipc_status.php, save_ipc.php, api/account/save_voucher.php similarly wrapped.
Test: tests/test_ipc_invoice_tx_cli.php — 11 checks, all 17 IPC/voucher/invoice suites pass.
FIX
Warehouse Delete — Cascade Made Atomic, Stock History Preserved
Bug: The delete cascade (product_stocks → stock_movements → locations → soft-delete) ran as 4 separate statements with no transaction — a mid-cascade failure left stock rows deleted but the warehouse still alive. It also hard-deleted stock_movements, destroying the stock in/out audit trail, while only soft-deleting the warehouse itself.
Fix: ajax_delete_warehouse.php and the duplicate delete branch in app/bms/stock/warehouses.php now cascade in one transaction; stock_movements is no longer deleted — history preserved.
Test: tests/test_warehouse_delete_tx_cli.php — 12 checks: live forced mid-cascade failure leaves everything untouched, live real cascade removes current state, keeps movement history, soft-deletes the warehouse.
FIX
Remaining Multi-Write Endpoints (Stock / CRM / Config) Made Atomic
Final batch — 9 endpoints with genuine multi-step writes and no transaction, now wrapped: api/rfq_quick_add_product.php, api/crm/move_lead_stage.php, bulk_update_leads.php, add_activity.php, edit_lead.php, manage_stage.php, api/pos/delete_salary_component.php, api/account/save_account.php, api/finance/manage_revenue_schema.php.
Also audited and deliberately left not wrapped: update-or-insert endpoints, per-row-reporting bulk import loops, and get-or-create lookups — a failure there leaves a valid reusable row, not corruption.
Test: tests/test_multiwrite_tx_coverage_cli.php — 22 checks, all 26 related CRM/account suites pass.
✓ Capstone: Financial Report Integrity Suite
tests/test_financial_reports_integrity_cli.php — 17 checks proving the four statutory reports stay correct after the whole sweep, and guarding them going forward (runs on every push via the pre-push hook): ledger invariant ΣDr = ΣCr, Balance Sheet balances (Assets = Liabilities + Equity), TB/P&L/Cash Flow internal consistency, and a live end-to-end proof — posts a real Dr Expense / Cr Asset entry, confirms P&L and TB move by exactly that amount, confirms every statement still balances, then removes the entry and re-proves the ledger is balanced.

Standalone Customer LPO Module — LPO → DN(Outbound) → Invoice Chain

PRs #1063, #1076 · 105+ test checks

LPO previously only existed as a modal-based tab inside a customer's detail page with no real approval workflow. Rebuilt to Purchase-Order parity as a full standalone module, wired all the way through to a working document chain: LPO → Delivery Note (Outbound) → Invoice. The Sales menu was reordered to match the natural document flow.

FEAT
LPO Foundation + Three-Approval Workflow
migrations/2026_07_01_lpo_standalone_foundation.php adds prepared/reviewed/approved actor+role+timestamp columns and project_id to customer_lpos; new permission page-key lpo (review/approve restricted to Admin + Managing Director).
New standalone pages mirroring the PO module: lpos.php, lpo_create.php, lpo_view.php, print_lpo.php — live product search, three-approval trail, matching print letterhead.
api/customer/save_lpo.php (unified save, no longer accepts an arbitrary status field), review_lpo.php, approve_lpo.php replace the deleted add_lpo.php/update_lpo.php.
FEAT
LPO → DN(Outbound) → Invoice Linkage
deliveries.party_type widened to include 'customer'; dn_outbound.php accepts ?lpo_id= to prefill remaining-to-deliver quantities and locks "Send To" to the LPO's customer.
api/approve_dn.php auto-advances the linked LPO to partially_fulfilled/fulfilled based on delivered-vs-ordered quantities (status only ever advances).
invoice_create.php accepts ?delivery= to prefill from an approved outbound DN, pricing from the matching LPO line falling back to products.selling_price; invoice view/print show "DN Ref:" / "LPO Ref:".
Sales menu reordered: Quotations → Sales Orders → LPO → DN(Outbound) → Invoices → POS → ... → Credit Notes.
FEAT
Warehouse Field Added to LPO and Invoice
Gap: neither customer_lpos nor invoices had a warehouse_id column, so their inventory-product stock figures always fell back to products.current_stock (not warehouse-specific).
Fix: added the field to both, project-filtered the same way every other module works — project selected → only that project's warehouses; no project → only unassigned warehouses. LPO always requires a warehouse (inventory-only document); Invoice requires one only outside Service Invoice mode, and only on create.
Test: tests/test_lpo_invoice_warehouse_cli.php — schema + wiring checks, live insert/round-trip proof (transaction-wrapped, rolled back).
tests/test_lpo_module_cli.php ALL PASS 105 / 105
Lint, migration content, three-approval gating, LPO↔DN↔Invoice wiring, menu order, live DB state, and a transaction-wrapped live create/review/approve round trip.

CRM Dashboard & Lead Import — Stabilization (5 Bugs)

PRs #1053, #1055, #1057 · 6 commits · 100/100 tests

Yesterday's CRM Professional Upgrade shipped nine feature phases; today closed out the rough edges that only surface once real users click through the dashboard and the CSV import wizard.

BUG 1
Dashboard Widgets Stuck on "Loading…" Forever
Bug: safeOutput() is not a global function — it only lived in pos_dashboard.php. crm_dashboard.php, crm_campaigns.php, and crm_leads.php all called it without defining it, throwing ReferenceError on every render and leaving Recent Leads, Due Activities, Top Performers, and the mobile card view stuck at "Loading…" permanently. KPI cards and charts were unaffected (no safeOutput calls there).
Fix: defined the 3-line XSS-escape helper at the top of each file's <script> block.
BUG 2
Dashboard Charts Blank — Chart.js Was Never Loaded
Bug: Chart.js was never loaded on the dashboard page. new Chart() threw ReferenceError, which silently aborted the entire $.getJSON callback — leaving all four charts blank and all three DataTables unpopulated, not just the charts.
Fix: added the Chart.js 4.4.0 CDN tag before the inline script block.
FEAT
All Three Dashboard Tables Converted to Proper DataTables
Recent Leads, Due/Overdue Activities, and Top Performers moved from raw HTML injection (.html()) to real DataTables with an S/No first column — gives sort, search, and pagination for free, and DataTables' own emptyTable message replaces the permanent "Loading…" when the API legitimately returns zero rows. Table headers also restyled from black to blue (table-primary) for visual consistency with the rest of the app.
BUG 3
CRM Import Leads — Page and API Both 404'd
Bug: crm/import_leads and api/crm/import_leads were never registered in roots.php — the page and API already existed from yesterday's Phase 8 work, only the route entries were missing.
Fix: registered both routes.
BUG 4, 5, 6
CSV Import Wizard — 3 Bugs Once It Was Reachable
Bug 1 — click opened no file dialog: #csvFile was nested inside #dropZone; a programmatic click on the hidden input bubbled back to the parent div and retriggered the handler, hitting RangeError: Maximum call stack. Fixed by moving the input above the drop zone and binding the click in JS instead of inline onclick.
Bug 2 — TypeError on empty CSV headers: the column auto-match loop called h.toLowerCase() with no null guard; added a h != null && guard before every keyword check.
Bug 3 — upload button stuck disabled after an error: an uncaught exception inside the $.ajax success callback skipped the complete handler. Wrapped the success body in try/catch so errors show a friendly message and the button always re-enables.
Bonus: drag-and-drop now updates the drop-zone visual on drop (previously only on file-picker select); dragleave resets the highlight.

Smaller Fixes — Payroll, Codes, DN Navigation, Deploy CI

PRs #1066, #1067, #1070, #1072, #1074 · 6 commits
Payroll Processing Failed for Every Employee (Production) PROD
  • Bug: yesterday's NSSF-employer feature made process_payroll.php write an 'employer_cost' breakdown line into payroll_items for every employee, but its migration only added the payroll.nssf_employer column — never added 'employer_cost' to payroll_items.item_type's ENUM. Every employee has nssf_employer > 0, so every "Process Payroll" run failed the whole batch with SQLSTATE[01000]: Data truncated for column 'item_type'.
  • Fix: migrations/2026_07_01_payroll_items_employer_cost.php adds the missing ENUM value (additive, idempotent). Checked for orphaned rows from prior failed attempts — none found. Test: tests/test_payroll_items_employer_cost_cli.php.
Company-Prefix Codes Never Re-Coded on Edit After Changing the Prefix FIX
  • Bug: codeForEdit() only recognised a code as "auto-generated, eligible for re-coding" if blank or matching the pre-prefix legacy pattern. A code already in PREFIX-TYPE-NNNN shape but stamped with a previous company prefix fell through to "manual code, leave alone" — so changing Company Profile's Document Code Prefix never propagated to existing records, even on edit. Shared by 16 edit endpoints (customers, suppliers, employees, quotations, sales orders, POs, vouchers, RFQ, DN, LPO, CRM leads, …).
  • Fix: added a second auto-generated shape — any prefix + TYPE + NNNN — so a stale-prefix code now re-codes to the current prefix on edit, same as blank/legacy codes always did. Genuinely custom codes are still left untouched. Test: tests/test_code_for_edit_stale_prefix_cli.php — proves exactly one sequence number burns on re-code, zero on already-current or genuinely-manual codes.
Outbound Delivery Note Tab/Button Leaked Into the Purchases View FIX
  • delivery_notes.php is shared by both menus — Purchases links to it plain, Sales links with ?type=outbound. The Outbound tab and its "Create DN (Outbound)" button rendered unconditionally, letting a Purchases user create sales delivery notes from the wrong screen. Follow-up fix caught the same leak in reverse: the "Record DN (Inbound)" button was showing on the Sales outbound view too.
  • Both wrapped in an $is_outbound_view check; Sales-side rendering (tabs, buttons, permission gate) left byte-for-byte identical.
Deploy CI Blocked Production After the LPO Module Merge PROD
  • Bug: deploy.yml's "Check critical API files exist" gate still listed api/customer/add_lpo.php / update_lpo.php — both intentionally deleted when the standalone LPO module replaced them with save_lpo.php. No test covers this hardcoded list, so it slipped through and failed the develop→main promotion deploy for PR #1065, blocking production.
  • Fix: removed the deleted files from the allowlist; added their replacements plus the new LPO pages and migrations so the gate reflects what the feature actually depends on.
Product Unit Field Editable Where It Should Be Locked FIX
  • A product's unit (kg, lt, pcs…) is set once at registration, but several line-item forms showed it as an independently-editable field after product selection. Scouted every Procurement/Sales form with live product search: Sales Order create/edit, Quotation form, GRN create all had it editable (DN Create, DN Outbound, Invoice Create were already correct). RFQ create is the one exception left editable by design (it allows free-text non-catalog requests) — but now locks once a real catalog product is linked.
  • Test: tests/test_unit_autofill_lock_cli.php — 24 checks.
