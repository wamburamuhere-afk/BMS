Financial Reports — Trial Balance + IS/CF Defaults Fixed

PR #1034 · MERGED · 1 commit
Trial Balance — Inactive Accounts Were Excluded FIX
  • Bug: glTrialBalance() in core/financial_reports.php joined accounts with an implicit status = 'active' filter. Accounts deactivated mid-year still had posted journal lines, but those lines disappeared from the Trial Balance completely, causing the report to appear unbalanced even though the ledger itself was sound.
  • Fix: Removed the status filter from the TB query. All accounts with journal activity appear, regardless of whether they are currently active or inactive. This matches standard accounting practice — a closed account's history must still be visible.
  • File: core/financial_reports.php
Income Statement & Cash Flow — Defaulted to Month-to-Date Instead of Year-to-Date FIX
  • Bug: The app/constant/reports/income_statement.php and cash_flow.php pages defaulted their date range to the current month (1st of this month → today). For most review sessions this showed too little data and confused managers who expected full-year figures.
  • Fix: Changed the default from date to 01 January [current year] so the report opens showing Year-to-Date figures. Users can still narrow the range manually.
  • Files: app/constant/reports/income_statement.php   app/constant/reports/cash_flow.php
✓ All three canonical financial reports now open showing meaningful data by default
Trial Balance: all-time (as-of today, including inactive accounts) · Income Statement: YTD (Jan 1 → today) · Cash Flow: YTD (Jan 1 → today)

Project Procurement — Context-Aware Navigation (RFQ → PO → GRN)

PR #1036 · MERGED · 8 commits · 5 files

A full procurement workflow lives inside each Project — RFQ → Purchase Order → GRN → Delivery Note. Previously, clicking into any of these from the project opened the external list and lost the project context. Back buttons navigated to the wrong pages. Dropdowns ignored project scope. All of these were fixed in this batch.

FEAT
RFQ — Context-Aware Navigation From Project
RFQ tab inside Project > Procurements now loads records scoped to the project.
"Create RFQ" button pre-fills the project; "View" / "Edit" links carry back=project&project_id=X in the URL.
Inside rfq.php (single RFQ view): "Back" returns to the project's Procurements tab instead of the global RFQ list.
Tab columns, action buttons, and mobile cards were made identical to the external rfq.php list — no missing columns, no stale action menu.
Files: app/bms/operations/project_view.php   app/bms/purchase/rfq.php
FEAT
Purchase Order — Context-Aware Create / Edit From Project
"Create PO" from within a project now opens purchase_order_create.php?project_id=X&back=project.
The PO create/edit page detects the back=project flag and shows "Back to Project" instead of "Back to Purchase Orders".
RFQ auto-fill: when a PO is opened from an RFQ, the supplier, items, and quantities are pre-populated automatically.
Warehouse fallback: if no warehouse is stored on the PO, the system defaults to the first active warehouse rather than showing an empty dropdown.
Files: app/bms/purchase/purchase_order_create.php
FEAT
GRN — Context-Aware Navigation in Project Details
"Create GRN" from the project's GRN tab now opens with project_id pre-set and a "Back to Project" link visible alongside "Back to GRNs".
GRN list row actions (View, Edit) carry the project context so navigation stays within the project.
File: app/bms/operations/project_view.php
FIX 1
GRN Supplier Dropdown Required a DN to Exist
Bug: grn_create.php built the supplier dropdown from the DN list — if no DNs existed for the project the dropdown was empty and the form could not be submitted.
Fix: Supplier dropdown now queries suppliers directly (scoped to project-assigned suppliers). DN remains an optional link, not a gate.
FIX 2
DN Items Showed Inflated Received Quantities
Bug: The pre-fill query for GRN line items summed received_qty across ALL GRNs that touched a delivery item, not just GRNs for the current DN. This caused items to appear "fully received" even when the current DN still had outstanding stock, hiding them from the GRN form via the HAVING pending_qty > 0 filter.
Fix: received_qty is now a correlated subquery scoped to the specific DN being received against. Each item's pending quantity is calculated correctly.
FIX 3
PO Dropdown Inside GRN Required an Approved DN
Bug: The PO dropdown in grn_create.php was only populated if a DN was already selected and approved — a circular dependency that prevented GRNs from being created without first going through a DN workflow.
Fix: PO dropdown now populates independently. DN selection remains optional; if chosen, it scopes the PO dropdown to that DN's purchase order.
FIX 4
DN PO Dropdown Showed All POs, Not Just Approved Ones
Bug: Inside the Delivery Note create/edit form, the "Link to PO" dropdown listed POs of any status (draft, rejected, etc.), allowing a DN to be linked to an unapproved order.
Fix: PO dropdown now filters status IN ('approved','ordered','partially_received') and, when inside a project, further scopes to that project's POs only.

POS — Hardware Barcode Scanner Support + 2 Bug Fixes

PR #1047 · MERGED · 4 commits · 73/73 tests

The POS terminal now supports USB and Bluetooth barcode scanners out of the box — no driver or plugin required. Scanners act as keyboards, firing all characters within <80ms followed by Enter. The implementation distinguishes scanner bursts from human typing using a timing window, and handles audio feedback, visual flash, cart increment, and focus recovery automatically.

FEAT
Hardware Barcode Scanner Interceptor — Timing-Based Detection
Hidden scan input (#hiddenScanInput) is always focused off-screen — scanner always has a target regardless of what the cashier clicked last.
80ms timing window (SCAN_MAX_MS = 80): characters arriving faster than human typing trigger a scan; slow keystrokes pass through to the user.
Modifier keys ignored (Ctrl/Alt/Meta); F-keys pass through to existing keyboard shortcuts (F1 search, F2 clear, F3 pay).
Focus recovery: hidden input regains focus after any modal closes and after clicking non-interactive areas — scanner is always ready.
window._posHandleScan exposed for browser console and CLI testing.
File: app/bms/pos/pos_scripts_new.php
FEAT
Audio + Visual Feedback on Every Scan
Web Audio API beep — no external library. Success: 880Hz square wave (80ms). Error: 200Hz sawtooth (300ms).
Header flash: POS header bar flashes green (#198754, 400ms) on a successful scan; red (#dc3545, 600ms) when the barcode is not found.
Non-blocking Bootstrap toast in the top-right corner: shows product name + price on success, or "Barcode not found" on error. Auto-dismisses after 3 seconds.
#scannerReadyBadge chip in the POS title bar confirms the scanner is active.
Files: app/bms/pos/pos.php   app/bms/pos/pos_scripts_new.php
FEAT
Cart Logic — Add or Increment, Never Duplicate
Scanning a barcode the first time adds a new cart line.
Re-scanning the same barcode increments the quantity on the existing line — no duplicate rows, no modal needed.
Lookup is case-insensitive and whitespace-trimmed so slightly dirty scanner reads still match.
BUG 1
Category Filter Broke Scanner — Products Outside Active Category Returned "Not Found"
Bug: When a cashier clicked a category button (e.g. "Beverages"), products[] was replaced with only that category's items. Scanning a product from any other category returned "not found" even though it existed in the system.
Fix: Added allProducts[] — a separate global that holds the full catalog and is only updated on unfiltered loads. handleBarcodeScanned() searches allProducts first so it always finds any product, regardless of the active category filter on the grid.
File: app/bms/pos/pos_scripts_new.php
BUG 2
Search Box Did Not Match Barcodes Despite Placeholder Saying It Would
Bug: The product search input had placeholder text "Search by name, SKU or barcode" but the API WHERE clause only checked product_name and sku. Typing or pasting a barcode into the search box returned no results.
Fix: Added OR p.barcode LIKE :search to the WHERE clause in simple_products.php.
File: api/pos/simple_products.php
PROD CRASH
GRN Create — ONLY_FULL_GROUP_BY Fatal Error on Production
Bug: grn_create.php line 144 grouped by di.delivery_item_id but referenced recv.received_qty (from a LEFT JOIN subquery) without an aggregate function. Local WAMP has ONLY_FULL_GROUP_BY disabled so it passed locally; production MySQL 5.7.5+ has it enabled by default and threw a fatal SQLSTATE[42000] error whenever a cashier clicked "Create GRN" from an approved Delivery Note.
Fix: Wrapped both uses in MAX() — COALESCE(MAX(recv.received_qty), 0) — in both the SELECT column and the GREATEST() expression. Result is mathematically identical since each group has exactly one recv row.
File: app/bms/grn/grn_create.php
