POS Dashboard — Full Rewrite + 4 Bug Fixes

PR #1024 · MERGED · 5 commits

The POS Dashboard was completely rewritten. The old design toggled between a "Dashboard" view and a "Sales History" view using buttons — confusing and the source of multiple JS bugs. The new design shows Sales History always at the top and the Dashboard always below, no toggle needed. A proper period filter was added with smart date pickers specific to each period type.

FEAT
POS Dashboard Rewrite — Sales History First, Period Filter, Toolbar
Sales History section always visible at top of page — no longer hidden behind a toggle.
Dashboard section always visible below it — toggle buttons removed entirely.
Period filter with 5 modes and smart date pickers per mode:
  • Daily → single date (defaults to today)
  • Weekly → pick any date; Mon–Sun range computed automatically and labelled
  • Monthly → month + year dropdowns (default: current month)
  • Quarterly → Q1–Q4 + year dropdowns
  • Yearly → year dropdown (default: current year, last 5 years listed)
Default period changed from Monthly to Yearly.
Copy / CSV / Print toolbar added above the sales table, matching suppliers.php style.
Error handling: loadDashboard() now has a .fail() handler — previously the page could show "Loading…" forever if the API failed.
ui-constants.md compliance: stat cards, modal headers, gear dropdown all follow the standard.
File: app/bms/pos/pos_dashboard.php   Test: 74 checks, all passing.
FIX 1
safeOutput Undefined + Period Filter Panel Fix
Bug: safeOutput() was called in JS but is not a global in this codebase — caused DataTable and dashboard widget JS errors on every page load.
Fix: Defined safeOutput() locally in pos_dashboard.php.
Also: getDateRange() and showFilterPanel() replaced the old setPeriodDates() which did not handle the new period-specific pickers. All Apply buttons unified via .apply-btn class delegation.
Tests: 88 checks, all passing.
FIX 2
Stat Cards Not Updating When Period Changed
Bug: getActivePeriod() read .active CSS class but the click handler only swapped btn-primary ↔ btn-outline-primary — .active was never set. So stat cards always used the initial Yearly range regardless of which period button was clicked.
Fix: Changed getActivePeriod() to read .btn-primary class instead of .active.
Also: Recent Sales and Low Stock converted to proper DataTables (S/NO column, pagination, sort). initDashboardTables() initialises once; loadDashboard() uses .clear().rows.add().draw() to refresh data without destroying and recreating the table.
Tests: 95 checks, all passing.
FIX 3
BMS Print Footer Missing in DataTables Print Window
Bug: DataTables Print opens a new browser window containing only the table — footer.php's .bms-print-footer div is never cloned there. Printed POS reports had no footer (no company name, no user name, no date).
Fix: Added a DataTables customize callback that appends the full BMS footer structure (user name, role, datetime, BJP Technologies copyright) with inline styles matching the @media print rules.
Also added PRINT_ROLE and PRINT_YEAR JS constants; corrected PRINT_USER to read from session at page-render time (it previously referenced $_print_username which is set in footer.php, after the JS block).
Tests: 98 checks, all passing (3 new checks).
FIX 4
Period Button Shows Double-Selection + Redundant Dashboard Reload
Bug 1: The "Yearly" period button had a hardcoded Bootstrap active class. The click handler never removed it, so after selecting any other period, both that period AND Yearly appeared selected (both filled blue) simultaneously.
Fix: Removed hardcoded active class from Yearly button — btn-primary alone provides the visual selection state.
Bug 2: voidSale() and openReturn() called loadDashboard() unconditionally on success, firing an extra AJAX request even when the Dashboard panel was hidden.
Fix: Guarded with if (!$('#paneDashboard').hasClass('d-none')).

Materials — Converted from Custom Manual Pagination to Real DataTable FIX
  • Materials was the most divergent: completely custom AJAX row-injection with manual pagination functions (procFilterMatTable(), procGoToMatPage()), a custom filter bar HTML block, and a custom pagination div. No DataTable at all.

Remaining 8 Tabs (Purchases, GRNs, Invoices, etc.) — Destroy+Reinit Pattern FIX

Bank Reconciliation — Complete 4-Phase Upgrade

PR #1029 · MERGED · 52 files · +10,959 / −82

The bank reconciliation system had two core problems: (1) book balance was read from accounts.current_balance which can drift from the actual GL, and (2) petty cash transactions never appeared in the bank register. Both fixed, plus adjusting-entry workflow, period overlap guard, beginning-balance chain, two-column print report, period lock, and unreconcile with audit trail.

Phase 1
Period-Aware Book Balance + Petty Cash in Bank Register
New function accountLedgerBalanceAsOf(PDO, $accountId, $asOf) in core/account_balance.php — computes true GL balance as of any date from journal_entry_items (posted only), replacing the stale accounts.current_balance.
create_reconciliation.php book_balance now calls this function with the period-end date.
get_bank_balance.php rewritten to return ledger-as-of balance when as_of param provided.
Petty cash: api/petty_cash/save_transaction.php calls recordBankTransaction() after each post; edit path calls reverseBankTransaction() first; delete path clears register rows by reference number.
Migration: migrations/2026_06_29_bank_recon_opening_balance.php — adds opening_balance column to bank_reconciliations.
Phase 2
Adjusting Journal Entries + Create-from-Statement-Line
New table bank_reconciliation_adjustments (migration 2026_06_29_bank_recon_adjustments_table.php, InnoDB, no FK to bank_reconciliations which is MyISAM).
New API api/account/add_reconciliation_adjustment.php: 6 types (bank_charge, interest_earned, NSF, standing_order, other_out, other_in). Posts balanced GL entry, writes bank register row, auto-matches, stores in adjustments table.
New API api/account/create_entry_from_statement_line.php: for transactions on bank statement not yet in books — creates GL entry and auto-matches the original line.
UI: "Add Adjustment" button + modal, "Create Entry" button on unmatched lines, Adjustments panel listing all adjustments with type/amount/memo.
Phase 3
Beginning-Balance Chain + Two-Column Print Report + Overlap Guard
Overlap guard: create_reconciliation.php rejects new reconciliations whose period overlaps an existing non-cancelled one for the same bank account.
Beginning-balance chain: opening_balance auto-populated from the prior finalized reconciliation's adjusted_balance — unbroken chain. If no prior, opening_balance = 0.
Worksheet exclusion: lines already cleared in a different reconciliation excluded from current worksheet.
Two-column print report: Bank Statement side (statement balance ± uncleared items = adjusted bank balance) vs Book side (book balance ± adjustments = adjusted book balance). "BALANCED" in green or "UNBALANCED" in red.
Phase 4
Period Lock + Unreconcile With Audit Trail
New core core/recon_period_lock.php — assertNotInFinalizedReconPeriod(PDO, $entryId): checks if any account touched by a journal entry has a finalized reconciliation covering that date; throws descriptive Exception if so.
Wired into 3 journal mutation endpoints: void_journal.php, reverse_journal.php, update_journal.php — all block if entry date falls in a locked period.
New API api/account/unreconcile.php (requires canApprove): reverts reconciliation → pending, reverts bank_transactions → cleared, logs full audit trail with mandatory reason (≥10 chars).
UI: "Unreconcile" button (canApprove only), SweetAlert2 confirmation, reason textarea.

GL / Financial Report Analysis — Why IS Shows Loss While BS Is Balanced

Research & Diagnosis

Full diagnostic investigation performed today. Both Income Statement and Balance Sheet read only journal_entries — the report code is correct. The problem is what gets posted there.

Root Cause — Revenue Almost Completely Missing from the GL
Only ~550M TZS of revenue has ever reached a revenue-category account in journal_entries (7 invoice approvals + 32 POS sales). All other revenue flows through recordGlobalTransaction() which mirrors to the GL as entity_type='books_transaction' — but the credit leg lands on account 2 (Opening Balance Equity, inactive, equity) NOT on any revenue account.
Why Balance Sheet Looks Positive While IS Shows a Loss
The mirror posts Dr account 13 (Marketing, inactive, expense) / Cr account 2 (Opening Balance Equity, inactive, equity). IS sees 12.5B expenses with zero revenue → loss. BS sees 12.5B equity credits → looks positive. Meanwhile 552 payroll accruals correctly post 151B salary expenses → IS shows massive loss, BS shows 151B in liabilities offset by equity.
Where Old Data Is Stored
recordGlobalTransaction() writes to: (1) transactions table (header); (2) books_transactions table (Dr/Cr legs with OLD pre-2024 account IDs); (3) journal_entries mirror using those same old IDs; (4) accounts.current_balance via applyAccountBalanceDelta(). The GL mirror exists but the account classifications are wrong.
Fix Path — F1 Legacy Dual-Write Cleanup (Next Phase)
Replace postInflow() / recordGlobalTransaction() in 8 remaining flows (IN-1/2/4, OUT-5/6/9/10/11) with direct postLedgerEntry() calls using correct new account codes (1-1200 AR, 4-1000 Sales Revenue, etc.). Revenue will finally land in a revenue account and the IS will show the real picture.
