1. Request for Quotation (RFQ) System

The RFQ module has undergone a complete architectural overhaul to support a rigid 3-stage authorization pipeline. This ensures financial compliance and prevents unauthorized procurement requests.

1.1 Sequential Authorization Logic

We transitioned the RFQ from a single-state entry to a multi-phase workflow. The system now enforces a strict sequence:

Workflow Stages:
  • Draft: Initial entry. Action: "Submit for Review".
  • In Review: Verification phase. Action: "Approve RFQ".
  • Approved: Final state. Document is locked and ready for distribution.

1.2 Immutable Audit Trail (Snapshots)

To ensure data integrity, we implemented a "Snapshot" system. Unlike traditional logging, snapshots capture the user's full name and role at the exact moment of authorization, storing them in dedicated columns:

Column Name Data Type Description
prepared_by_nameVARCHAR(150)Name of the user who created the RFQ.
reviewed_by_nameVARCHAR(150)Name of the user who verified the RFQ.
approved_by_nameVARCHAR(150)Name of the user who gave final approval.

1.3 Professional Print Redesign

The print_rfq.php layout was redesigned to match international procurement standards. The legacy "Authorized Signature" block was replaced by a triple-signature panel featuring detailed metadata for every stage of the approval process.

2. Purchase Order & Delivery Note Optimization

The Purchase Order (PO) and Delivery Note (DN) modules were synchronized with the new RFQ workflow to provide a unified experience across the Procurement lifecycle.

2.1 Purchase Order (PO) Stabilization

A critical bug was identified where the PO status remained "Ordered" instead of "Approved" after final authorization. This was resolved in update_purchase_order_status.php and approve_purchase_order.php. Additionally, the PO details view now includes a real-time "Authorization History" panel.

2.2 Delivery Note (DN) Schema Migration

During deployment to the online server, we identified missing columns in the deliveries table. A comprehensive migration script was developed to resolve the SQLSTATE[42S22] errors.

Key Schema Updates (Table: deliveries):
  • Added do_id (Foreign Key for Delivery Orders).
  • Added purchase_order_id (Direct link to PO).
  • Implemented the 3-stage snapshot columns (Prepared/Reviewed/Approved).
  • Extended status ENUM to include 'review'.

2.3 Standardized DN Workflow

The Delivery Note creation process (dn_create.php) now utilizes a sophisticated supplier filter. It only displays suppliers with active, approved Purchase Orders, ensuring that goods can only be received against authorized contracts.

2.4 Professional DN Print Layout

The print_delivery_note.php script was updated to match the PO print layout. This includes the same high-resolution signature table, ensuring that the entire document trail looks uniform and professional.

3. System Security & Quality Assurance

To support these major changes, we enhanced the underlying system security and testing framework.

3.1 Environment Synchronization (Migration Framework)

We introduced a URL-based migration framework allowing the administrator to sync the online database with local schema changes securely. These scripts are protected by a unique token-based authentication system (bms_migrate_2024), bypassing standard session requirements for automated deployment.

3.2 Automated Integration Testing

A new suite of integration tests was developed to prevent future regressions:

  • api/test_rfq_workflow.php: Validates the full RFQ lifecycle.
  • api/test_po_workflow.php: Verifies PO status transitions and math calculations.
  • api/test_dn_workflow.php: Ensures DN creation and snapshot accuracy.

3.3 Conclusion

The BMS system is now significantly more robust, compliant, and professional. The procurement workflow is fully automated and secured by a traceable audit trail, making it ready for high-volume business operations.
